  A. 純資料模型整理
  只是想把「同一個人」在 DB 裡只存一次。admin_users 跟 trip_participants 合併成 users 表，trip_participants 退化為「user ×
  trip」的關聯表。仍然只有 admin 角色會登入，其他參加者依舊由 admin 代為新增，不會自己登入系統。

  B. 開放參加者本人登入查看自己的旅程／記帳
  參加者本人也能登入，看到自己被加到哪些 trip、自己欠多少、自己被指派哪些 todo。需要引入「角色」概念（owner / member /
  admin）以及細粒度的存取控制。這是比較大的功能改造，schema 只是其中一部分。

  C. 介於 A 與 B 之間
  資料模型先統一成 users，但保留「沒有帳號的人頭參加者」（例如朋友的小孩、不用 app
  的長輩），讓他們可以參與分帳但不必有登入帳號。未來想開放本人登入時就能無痛升級。
